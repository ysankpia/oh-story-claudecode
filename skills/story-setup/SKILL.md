# story-setup：Claude/Codex/Droid 写作基础设施部署

你是写作基础设施部署器。只向用户项目部署 Claude Code、Codex、Droid 或其组合所需的 hooks、agents、规则和参考资料。

**执行铁律：不覆盖用户已有配置，合并而非替换；不删除用户项目中的未知目录或旧工具配置。**

## 支持边界

- 合法 `target_cli` 是 `claude-code`、`codex`、`droid` 的非空组合；多端值按此顺序用逗号连接。
- 14 个 story skill 保持完整；本 skill 不复制一套通用 skills 文件作为其他运行时的替代安装方式。
- 检测到旧版部署遗留的非三端 adapter 标记时，报告“该 adapter 已停止支持”，要求用户明确选择当前支持目标；不得修改或删除这些遗留文件。
- 自定义 agent 未部署、当前会话不能 spawn，或运行时未暴露对应 registry 时，相关写作/审稿流程仍必须走 `solo/direct` fallback。

## Phase 1：检测与选择

1. 检查 `.story-deployed`：
   - `agents_version` 缺失、非整数或小于 `23`：标记为待更新。
   - `agents_version: 23`：确认是否重新部署。
   - `agents_version` 大于 `23`：停止，提示先更新本 skill，避免旧版本覆盖新部署。
2. 检查书名目录（含 `追踪/` 或用户明确指定的项目目录）和 `.active-book`，只用于展示当前写作状态。
3. 检查 `.claude/`、`.codex/`、`.factory/`、`CLAUDE.md`、`AGENTS.md` 和各端 hook 注册：
   - 只识别 Claude 标记：默认候选 `claude-code`。
   - 只识别 Codex 标记：默认候选 `codex`。
   - 只识别 Droid 标记：默认候选 `droid`。
   - 多类存在：按 `claude-code,codex,droid` 顺序组合默认候选。
   - 都不存在：让用户选择一个或多个支持目标。
4. 如发现旧 adapter 标记，先说明它们不会被自动清理；仍以用户本轮选择的目标继续，不把旧标记写入新的 `.story-deployed`。

## Claude Code 部署

1. 读取并按 marker 合并 `references/templates/CLAUDE.md.tmpl` 到根 `CLAUDE.md`。
2. 递归复制完整目录树 `references/templates/hooks/` 到 `.claude/hooks/`，保留 `lib/common.sh`、`lib/sentinel.sh`、`story_hook_core.js` 和 `story_hook_cli.js`；对顶层 shell hooks 设置执行权限。
3. 复制规则、7 个 agent 和 agent references。agent 引用必须使用项目内 `.claude/skills/story-setup/references/agent-references/` 路径。
4. 将 `settings-hooks.json` 的 managed hook registrations 合并进 `.claude/settings.local.json`，保留用户 hooks 与未知字段。
5. 报告用户需要新开 Claude Code 会话，才能加载更新后的 agents。

## Codex 部署

1. 合并 `references/codex/AGENTS.md.tmpl` 到根 `AGENTS.md`。
2. 复制 `references/codex/agents/*.toml` 到 `.codex/agents/`；每个 TOML 必须含 `name`、`description` 和 `developer_instructions`。只读职责 agent 必须保留 `sandbox_mode = "read-only"`。
3. 复制 Codex hooks，并通过 `scripts/merge-codex-hooks.py` 合并 `.codex/hooks.json`：替换本 skill 管理的 registrations，保留用户 hooks 和未知顶层字段。
4. 复制 agent references 到 `.codex/skills/story-setup/references/agent-references/`。
5. 报告用户需要 trust 项目 `.codex/` 层、在 `/hooks` 审核 hooks，并新开 Codex 会话。运行时没有 custom-agent registry 时必须使用 `solo/direct`。

## Droid 部署

1. 合并 `references/droid/AGENTS.md.tmpl` 到根 `AGENTS.md`，保留 Codex 与用户自有段落。
2. 运行本 skill 自带的 `scripts/generate-droid-agents.py --dest {项目根}/.factory/droids`，从 Claude agent 模板确定性生成 7 个 Factory custom droids；模型使用 `inherit`，工具名映射为 Factory 原生名称。
3. 通过 `scripts/merge-droid-hooks.py` 合并 `references/droid/hooks/hooks.json` 到 `.factory/hooks.json`，只替换本 skill 管理的注册，保留用户 hooks 与未知字段。复制 launcher，并将共享 Python hook 部署为 `.factory/hooks/story_droid_hook.py`。
4. 复制 agent references 到 `.factory/skills/story-setup/references/agent-references/`。custom droid 只能读取这一条 canonical 路径。
5. 长拆文优先使用 Task 的 background 模式并用 TaskOutput 收集结果；每批落盘 `_progress.md` 后再启动下一批。会话中断时按文件断点恢复。
6. 报告用户在 `/hooks` 审核项目 hooks，并新开 Droid 会话以刷新 skills 与 droids。

## 共享 hook 契约

- Claude 的 Node core `references/templates/hooks/story_hook_core.js` 是正文网、字数、路径抽取、提交识别和连续性检查的唯一 JavaScript 权威实现。
- Codex 的 `references/codex/hooks/story_codex_hook.py` 保持同一对外行为；由回归测试验证正文网、连续性、提交提示和大纲守卫的结果一致。
- Droid 复用同一 Python hook 实现，只增加 Factory 事件注册和 launcher；不复制正文规则实现。

### Step 1：创建部署标记

部署完成后写入以下字段：

```yaml
deployed_at: 2026-07-19T00:00:00Z
agents_version: 23
setup_skill_version: 1.6.0
target_cli: claude-code | codex | droid | comma-separated combination
resolver_strategy: project-local
references_dir: selected adapters' canonical reference paths, comma-separated
```

`target_cli` 为多端组合时，`references_dir` 用逗号分隔各端项目内路径。旧 sentinel 中已停止支持的值只用于给出迁移提示，不能作为新部署目标。

## 验证安装

### Claude Code

- `.claude/hooks/` 包含完整 hook 树，`node --check` 能解析 `story_hook_core.js` 与 `story_hook_cli.js`。
- `.claude/agents/` 有 7 个 agent，`.claude/rules/` 每个文件有 `paths` frontmatter。
- `.claude/settings.local.json` 有效，managed hook commands 不重复。
- `.claude/skills/story-setup/references/agent-references/` 的每个被引用文件都存在。

### Codex

- `.agents/skills` 保持指向 `../skills` 的相对 symlink，仓库发现 14 个 skill。
- `.codex/agents/` 的 7 个 TOML 均可解析；`.codex/hooks.json` 有当前 6 个 managed registrations，用户 hooks 未被移除。
- `.codex/hooks/` 同时包含 Python、shell 和 Windows launcher；Python/Node 源文件均可解析。
- `.codex/skills/story-setup/references/agent-references/` 的每个被引用文件都存在。

### Droid

- 仓库 `.factory/skills` 指向 `../skills`，可发现恰好 14 个 skill。
- `.factory/droids/` 有 7 个可解析定义，均含 `name`、`description`、`model: inherit` 和合法 Factory tools。
- `.factory/hooks.json` 仅注册 Factory 支持的事件，所有命令经 `$FACTORY_PROJECT_DIR/.factory/hooks/run-story-droid-hook.sh` 启动。
- `.factory/skills/story-setup/references/agent-references/` 的每个被引用文件都存在。

### 多端组合

- 多套目录同时存在时，`CLAUDE.md` 与 `AGENTS.md` 的 managed section 不互相覆盖。
- `.story-deployed` 的 `target_cli` 按固定顺序记录所选端，全部 reference 路径均可读。

## 输出安装报告

报告必须说明：实际 `target_cli`、写入或合并的路径、保留的用户配置、需要重新打开的会话，以及任何不支持的旧 adapter 标记。不得声称已删除用户项目中的旧配置。
